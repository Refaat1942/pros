## ١. نطاق النظام وبيئة التشغيل المحمية

*Project Scope*

النظام عبارة عن حل برمجي متكامل لإدارة الأنشطة الإدارية، الطبية، والمخزنية بشكل مغلق تماماً. تم تصميم بيئة التشغيل لتعمل محلياً بنسبة 100% **(Offline)** على خادم داخلي داخل جدران المركز، مما يضمن الحفاظ الكامل على خصوصية وسجلات المرضى ويمنع خروج أي بيانات طبية إلى الإنترنت.

> **ℹ️ ملاحظة:** يعتمد النظام على معالجة البيانات داخلياً دون أي خدمات سحابية خارجية، مما يجعله مستقلاً ومقاوماً لأي انقطاع في الاتصال الخارجي، ويضمن سرعة استجابة فائقة **(Zero Latency)** داخل الشبكة المحلية.

> **⚠️ تحديد المسؤوليات والحدود التقنية المشتركة:**  
> تنحصر مسؤوليتنا البرمجية في بناء الأكواد اللوجيكية، هيكلة قواعد البيانات، وتجهيز واجهات الاستخدام. أما البنية التحتية (السيرفر المحلي، الشبكة الداخلية LAN، عناوين IP الثابتة) فتقع مسؤوليتها الكاملة على **مهندس تقنية المعلومات** التابع للمركز.

### الموديولات الأساسية للنظام (Smart Prosthetics ERP)

النظام منظومة ERP تعتمد محورياً على **QR Code و Barcode** لتقليل التدخل البشري وضمان تتبّع دقيق لكل حالة وكل قطعة مخزنية:

| الموديول الأساسي | الوظيفة المحورية | التقنية |
|---|---|---|
| **إدارة المرضى والجهات (EMR)** | تسجيل وتصنيف المرضى (مدني/عسكري) + شجرة الجهات المتعاقدة + التاريخ الطبي والقياسات والأشعة | `RDBMS علائقية` |
| **محرك التتبّع والتدفق** | تحويل الطلبات آلياً (Event-Driven) ومتابعة زمن التنفيذ (SLA) لكل قسم | `Dynamic QR Code Engine` |
| **إدارة المخزون والواردات** | التحكم بالمواد الخام والأجزاء الجاهزة ومنع الصرف العشوائي | `Barcode 1D/2D Serialized` |
| **محرك الحسابات والتسعير** | تسعير الحالات بخوارزميات WAC + أعلى سعر شراء | `Financial Background Automation` |

> **🔒 الفصل الكامل بين المسارات:** عزل تام في البيانات والمعالجات بين فئتي المرضى **المدنيين والعسكريين** لتلبية المتطلبات السيادية والأمنية لكل فئة.

## ٢. مسارات تفصيلية لحركة وتدفق العمليات

*System Workflows*

### مرحلة الاستقبال والتهيئة الرقمية (مشتركة)

1. يمسح موظف الاستقبال وثيقة الهوية ويولّد **رقم تعريف فريد وثابت للمريض (Patient ID)**.
2. يحدد **تصنيف المريض (مدني / عسكري)** — وهو ما يوجّه الحالة للمسار المناسب.
3. يربط المريض بجهة التعاقد مع نسبة التغطية وشروط الدفع.
4. يولّد النظام ويطبع **بطاقة تحتوي على Dynamic QR Code** فريد خاص بالمريض.
5. يمكن للمريض مسح الكود لمعرفة حالة الطلب، عدد الحالات في الطابور، والموعد المتوقع للتسليم.

### المسار المدني

1. الاستقبال والعيادة + QR
2. التوصيف والمعدلات (أكواد فقط)
3. محرك التسعير الآلي في الخلفية
4. موافقة الأدمن (حوكمة داخلية)
5. عرض السعر + انتظار موافقة الجهة
6. ربط الموافقة → مكتب التشغيل
7. صرف الخامات بالباركود + التصنيع
8. تركيب + مسح QR الختامي + فاتورة

### المسار العسكري

1. الاستقبال + رتبة + جهة سيادية + QR
2. التوصيف الفني (أكواد وكميات)
3. تقييم مالي صامت في الخلفية
4. تخطّي العرض والموافقة المالية
5. إرسال مباشر لمكتب التشغيل
6. صرف + تصنيع بالباركود
7. تسليم + إغلاق الملف
8. ترحيل التكلفة → ديون سيادية

### مسار الحالة الكامل (Status Pipeline)

```
استقبال (تصنيف + QR) → كشف وتشخيص → توصيف فني → تسعير آلي → مكتب التشغيل → تحت التصنيع → جاهز للتسليم → حالة مغلقة
```

### محرك التدفق المؤتمت (Event-Driven Workflow Engine)

| الحدث المحرك | الانتقال التلقائي | الإجراء الخلفي |
|---|---|---|
| اعتماد الطبيب للكشف والروشتة | ← انتظار التوصيف الفني | إدراج في قائمة مهام فنيي التوصيف |
| حفظ الخامات والمعدلات | ← التسعير التلقائي | استدعاء موديول الحسابات |
| انتهاء التسعير (مدني) | ← انتظار موافقة الجهة | توليد عرض سعر PDF + تجميد الطلب |
| انتهاء التسعير (عسكري) | ← جاهز للتشغيل مباشرة | تخطّي الحجز المالي |
| مسح QR خطاب الموافقة (مدني) | ← جاهز للتشغيل | فك التجميد + توليد رقم أمر التشغيل |
| مسح باركود صرف المواد | ← تحت التصنيع | تحديث أرصدة المخازن + إشعار الورش |
| تأكيد الفني إنهاء الصناعة | ← جاهز للتسليم | تحديث الحالة — BOM تام |
| مسح QR بطاقة المريض عند التسليم | ← حالة مغلقة | أرشفة الملف + ترحيل القيود المالية |

### مراحل التصنيع الفرعية (Manufacturing Sub-Stages)

| المفتاح | التسمية | العلاقة بـ BOM |
|---|---|---|
| `issue` | صرف بالباركود | BOM **خام** ← يُصرف بمسح الباركود |
| `generation` | توليد | BOM **تحت التشغيل** |
| `assembly` | تجميع | BOM **تحت التشغيل** |
| `casting` | صب | BOM **تحت التشغيل** |
| `finishing` | تشطيب وفحص | BOM **تام** — جاهز للتسليم |
| *تسليم* | حالة مغلقة | تُغلق بمسح QR الختامي |

> **📌 تمييز مهم:** BOM «تام» = اكتمال تصنيع الطرف. «حالة مغلقة» = تسليم الطرف وإقفال الحساب المالي — **خطوتان منفصلتان تماماً**.

## ٣. بنية الوصول وفصل لوحات الأدوار

*Dashboard Access — بدون Role/Permissions معقدة*

يحتوي النظام على فصل وصول تشغيلي بسيط: كل موظف يسجّل الدخول عبر جارد الـ `web` الافتراضي، ويُوجَّه إلى **لوحة واحدة** حسب `users.role_id` → `roles.slug`. **لا يوجد** نظام Permissions أو Spatie — الإدارة لا تدير أدوار/صلاحيات دقيقة؛ الفصل يتم على مستوى اللوحة والـ Services (إخفاء الأسعار/الأرصدة حسب الدور):

| الدور التشغيلي | الصلاحيات الوظيفية | المستوى المالي/المخزني |
|---|---|---|
| **إدارة النظام (المالك)** | تحكم مطلق: إعدادات النظام، حسابات الموظفين، فواتير الشراء، بيانات الموردين، Audit Trail | صلاحية شاملة غير محدودة لكل التقارير والأسعار |
| **الطبيب المعالج** | البيانات الطبية والتشخيصية للمرضى، السجل الطبي القديم، التوصيات الطبية | محجوب مالياً بالكامل — لا يرى أي أسعار |
| **القسم الفني (الأخصائي)** | شاشات التوصيف الفني: أكواد وكميات مطلوبة فقط | محجوب مالياً + محجوب رصيد الكميات المتوفرة |
| **موظف الاستقبال** | تسجيل بيانات المرضى، تقويم المواعيد، إصدار عروض الأسعار، مسح QR الموافقات | قيم عروض الأسعار فقط — لا أسعار شراء أو أرباح |
| **أمين المخزن** | أوامر التحضير، إدارة BOM، صرف الخامات، الجرد، المشتريات المستلمة | محجوب مالياً بالكامل — كميات ومواصفات فقط |

## ٤. اللوجيك الذكي للمخازن والسياسة السعرية

*Barcode + WAC + Highest Purchase Price*

### السياسة المالية المزدوجة (Dual Costing)

| الغرض | الطريقة | المعادلة / القاعدة |
|---|---|---|
| **تقييم المخزون والتقارير** | **WAC** متوافق مع IFRS | `(قيمة الرصيد السابق + قيمة الوارد) ÷ (كمية الرصيد + كمية الوارد)` |
| **تسعير الحالات وعروض الأسعار** | **أعلى سعر شراء** | أعلى سعر دفعة كميتها `> 0` — يتجاهل المتوسط |

> **💡** عند التسعير يتجاهل النظام WAC ويستخدم **أعلى سعر شراء** كقاعدة لحماية المركز من تآكل رأس المال الناجم عن فروق الأسعار والتضخم عند إعادة الشراء مستقبلاً.

### قائمة المواد (BOM) — دورة الحياة الكاملة

| مرحلة BOM | المعنى التشغيلي | حركة المخزون | من ينفّذ |
|---|---|---|---|
| **خام (raw)** | قائمة مواد معتمدة — بانتظار الصرف | لا خصم بعد | تُنشأ تلقائياً عند تأكيد الموافقة |
| **تحت التشغيل (WIP)** | خامات صُرفت للورشة وجاري التصنيع | خصم من StockCatalog | أمين المخزن — زر «صرف للورشة» |
| **تام (finished)** | منتج تام الصنع — جاهز للتسليم | لا إرجاع؛ BOM مغلقة | أمين المخزن — زر «إغلاق — تام» |

### الوحدات البرمجية المشتركة

| الوحدة | الملف | الدور |
|---|---|---|
| `CasesWorkflow` | `cases-workflow.js` | مسار الحالة + تصنيف + Patient ID/QR + رقم أمر التشغيل |
| `OperationsDesk` | `operations-desk.js` | مكتب التشغيل: استقبال المسارين + توليد أمر التشغيل المركزي |
| `BomInventory` | `bom-inventory.js` | BOM: إنشاء، صرف بالباركود، إغلاق، `canDeliver` |
| `StockCatalog` | `stock-catalog.js` | بطاقة الصنف + باركود + WAC + أعلى سعر |
| `PricingQueue` | `pricing-queue.js` | طابور التسعير + اعتماد الأدمن + `highestUnitPrice` |

## ٥. تقنيات الأتمتة المحلية والمعاملات المحاسبية

*Local Automation Technologies*

- **🔍 أتمتة مستندات الموافقات (Local OCR):** محرك قراءة ضوئية محلي يسحب صور الموافقات وفواتير الشراء ويحولها لبيانات رقمية في قاعدة البيانات بشكل آلي.
- **📱 تتبع وتأمين الوثائق محلياً (Local QR):** توليد QR Codes فريدة عند طباعة عروض الأسعار، ومسحها والتحقق من سلامة الطلب في أقل من ثانية.
- **💰 منظومة التقارير والمديونيات التراكمية:** ترحيل القيود وتحديث الحساب الجاري لكل جهة فور تأكيد صرف الخامات، مع خصم التحصيلات ونسب التحمل.
- **📊 تحليلات المخزون الذكية:** Dashboard يكشف الأصناف الراكدة لتفادي تجميد السيولة، والأصناف الأكثر طلباً لتأمين توفرها قبل نفادها.
- **🛡️ سجل العمليات والرقابة التلقائية (Audit Trail):** تسجيل كافة الحركات مع معرف الموظف، نوع الحركة، طابع الوقت بالثانية، وحالة البيانات قبل وبعد.

## ٥٫٦. معايير الأمان وسجل التدقيق الحصين

*Immutable Audit Log*

يُدعَّم النظام بسجل تدقيق يلتقط **كل نقرة زر أو تعديل** فورياً وتلقائياً دون قدرة أي مستخدم على تعطيله. يحفظ لكل حركة:

| الحقل | القيمة |
|---|---|
| **هوية المستخدم** | User ID / Username |
| **التوقيت الدقيق** | Timestamp بالثانية |
| **معرف الجهاز** | IP / MAC Address |
| **نوع العملية** | إنشاء / تعديل / حذف |
| **حالة البيانات** | Before / After Payload |
| **نوع السجل** | Append-Only جداول |

> **⚠️ قاعدة غير قابلة للاختراق:** السجل مبني على جداول Append-Only. لا يملك أي مستخدم — بما في ذلك مدير IT أو المدير العام — أي صلاحية لتعديل أو حذف أي سطر، مما يجعله **وثيقة إثبات قانونية معتمدة**.

## ٦. حالة التنفيذ

*النموذج الأولي HTML/JS مقابل مرحلة الإنتاج Laravel*

### ✅ منفّذ في النموذج (يونيو 2026)

- ✅ تصنيف المريض (مدني/عسكري) + عزل المسارين — `CasesWorkflow`
- ✅ Patient ID ثابت + بطاقة QR + خدمة ذاتية للحالة — الاستقبال
- ✅ مكتب التشغيل + رقم أمر التشغيل المركزي — `OperationsDesk`
- ✅ موافقة الأدمن + موافقة الجهة + تقييم صامت عسكري — `CasesWorkflow`
- ✅ WAC + أعلى سعر شراء (تسعير مزدوج) — `StockCatalog`
- ✅ صرف بالباركود + إنذار عدم تطابق — المخزون
- ✅ بطاقة صنف + رصيد أول المدة + حركة وارد + إعادة حساب WAC — `StockCatalog`
- ✅ 5 لوحات قيادة BI بمؤشراتها الكاملة — Admin Dashboard
- ✅ EMR + روشتة + عيادة صامتة — Doctor Dashboard
- ✅ سجل تدقيق حصين (User/IP/MAC/Before-After) — محاكاة

### ⏳ متبقٍ لمرحلة الإنتاج (Laravel Backend)

| # | البند | ملاحظة |
|---|---|---|
| ١ | حجز `reserved` عند BOM خام | Prototype لا يحجز الكمية قبل الصرف |
| ٢ | ترحيل مالي فعلي عند الصرف | القيود والمديونيات — mock في الإدارة فقط |
| ٣ | OCR حقيقي + باركود فعلي (Handheld) | Prototype: محاكاة؛ الإنتاج: محرك OCR |
| ٤ | Audit Log حصين فعلي | جداول Append-Only + Middleware بـ IP/MAC |
| ٥ | مصادقة + ربط Guard بلوحة الدور | `auth:web` + `role_id` → dashboard route (بدون permissions) |
| ٦ | قاعدة بيانات | استبدال LocalStorage بـ MySQL/PostgreSQL |

> **✅ ملخص الحالة:** النموذج الأولي **جاهز للعرض والموافقة** على التدفق والواجهات بما فيها التصنيف المدني/العسكري، مكتب التشغيل، الباركود، وWAC. مرحلة Laravel تنقل نفس اللوجيك إلى Backend مع مصادقة بسيطة (لوحة لكل دور)، OCR، باركود فعلي، وترحيل محاسبي حقيقي.
